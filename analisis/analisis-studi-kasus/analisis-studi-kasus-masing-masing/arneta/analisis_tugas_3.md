## Analisis Tugas 3 - The Enterprise Digital City (Service A)

- Transaksi penting di Service A adalah "Pencatatan Mahasiswa Baru" pas lagi nembak API POST /api/v1/mahasiswa. kenapa menurut saya dia itu penting karena mendaftarkan mahasiswa baru bukan transaksi yang biasa aja dia itu punya efek dmoino ke sistem lain, karena transaksi ini membutuhkan SOAP dan RabbitMQ. 
- kenapa harus pake SOAP: kaya ini tuh contoh nya kaya "Buku Laporan Resmi". tujuannya untuk kepatuhan, misal nih pemerintah atau pihak universitas pusat harus tau setiap kali ada mahasiswa baru yang terdaftar, ini untuk mencegah adanya "mahasiswa anomali" yang memanipulasi data di database lokal. trs dia tuh harus pake SOAP karena kan di sistem pemerintahan, sistem audit versi lama biasanya kan di bangun menggunakan teknologi SOAP(XML), SOAP sangat kaku, aman dan memiliki aturan penulisan yang ketat. tapi alur nyatanya ketika maba masuk ke database kita, aplikasi kita wajib langsung membuat laporan berformat XML yang isi nya:
    - siapa tim yang mendaftarkan (team-9)
    - apa nama aktivitas nya (Maba)
    - detail datanya 
yang terus nanti tuh XML nya bakalan di krim ke server pusat. jika server pusat nya setuju, mereka akan memberikan ReceiptNumber. nah si ini lah yang menjadi bukti kalo pendaftaran mahasiswa baru itu udah di audit oleh pusat dan legal.
- kenapa harus disebarkan pake RabbitMQ? soalnya gambarannya nih si RabbitMQ itu adalah cara kita ngobrol ke service b dan c. tujuannya kalo ke service b itu biar tau kalo ada maba yang namanya "ana" agar dia tuh bisa memvalidasi saat ana mau mengambil mata kuliah (KRS). kalo ke service c itu, biar dosen bisa meng input nilai ke ana nanti. kenapa pake RabbitMQ lebih bagus,jadi kalo ada API service B dan C satu per satu lengsung itu kan proses nya bakalan lambat banget trs kalo misal service b nya mati, pendaftaran maba nya bakalan langsung ikut error. nah kalo pake RabbitMQ ini, kita tinggal ngasih pesan pake kunci mahasiswa.created ke RabbitMQ, dan tugas kita tuh bakalan kelar, dan service B dan service C nya bakalan mengambil pesan tersebut dari RabbitMQ kapan pun mereka siap, bahkan jika mereka sempat mati sementara.
- kesimpulannya: transaksi mencatat maba itu krusiap, kita butuh SOAP untuk meminta restu dan bukti legal dari server pusat berupa ReciptNumber, dan kita butuh RabbitMQ untuk mengumumkan ke layanan lain (KRS & Nilai) secara cepat dan aman agar merekatahu kalo ada maba tanpa mengganggu kinerja server kita.

- penjelasannya : 

Client/admin mengirimkan perintah unutk menambah mahasiswa baru lwat API Gateway dengan membawa API Key nya sendiri (KEY-MHS-233), trs si API Gateway nya itu meneruskan nnya ke MahasiswaController, nah di MahasiswaController ini ada data maba nya kaya (NIM, nama, EMAIL), nah dia tuh bakalan di validasi dulu, klo data nya valid dia bakalan nyuruh Entity Mahasiswa untuk menyimpan data nya ke database dan database nya akan memberikan respons kalo udah di simpen.
Karena mau menyambungkannya ke Cloud Dosen jadi MahasiwaController harus punya akses masuk dengan mengirimkan request login M2M ke SSO Server menggunakan API Key kelompok 09, dan nanti tuh si SSO Server nya bakalan memverifikasi API nya dan mengirimkan balik Token M2M JWT khusus untuk Team-09.
Trs MahasiswaController nya itu akan memanggil SoapAuditService untuk mengurus audit resmi nya, di SoapAuditService itu data mahasiswa yang tadi akan di konversi ke format XML, nah trs dia bakalan menembak server SOAP Audit Server di pusat menggunakan token M2M tadi. Nah di bagian ini ada dua kemungkinan, klo respon SOAP nya sukses dia bakalan ngasih ReceiptNumber trs si SoapAuditServices nya itu akan mengekstrak nomor nya trs nyuruh Entitiy AuditLog untuk menyimpan log dengan status success dan receipt_number nya ke database. Tapi klo misal respon SOAP nya gagal atau timeout, dia bakalan nyuruh Entity AuditLog buat nyimpen log dengan status failed dan receipt_number nya di set null, trs dia melapor balik ke MahasiswaController.
Trs Mahasiswa Controller nya akan memanggil RabbitMQ untuk menyebarkan informasi ke service B dan C, nah si RabbitMQServices nya ini akan menyusun payload event JSON dengan routing key mahasiswa.created. Trs RabbitMQService nya akan megirim pesan tersebut ke RabbitMQ Broker terpusat. Nah di sini juga ada dua kemungkinan, klo publish RabbitMQ nya sukses dia bakalan ngasih respon sukses dan ngirim status terkirim ke controller, tapi klo publish nya gagal dia bakalan ngirim status gagal ke controller.
Setelah semua proses database lokal, SOAP audit dan RabbitMQ nya beres, MahasiswaController mengirimkan pesan sukses 201 Created ke API Gateway, dan nanti si API Gateway nya itu bakalan ngasih responnya balik ke client/admin.
